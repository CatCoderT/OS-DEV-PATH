What is Assembly?

C code gets compiled into machine code — raw numbers the CPU understands. Assembly (ASM) is one step above that. Instead of numbers, you write human-readable instructions that map almost 1:1 to what the CPU actually does.

In C you write x = y + 5. In ASM you write the exact CPU steps to do that. It feels like talking directly to the hardware.

examplex86_64 NASM
; This is what   x = y + 5   looks like in ASM
mov  rax, [rbx]       ; load y into rax
add  rax, 5          ; add 5 to rax
mov  [rcx], rax      ; store result into x
Good news: Because you know C, ASM will feel familiar. You already understand variables, functions, loops, and memory. ASM is just doing all of that manually.
Chapter 01

Why an OS Needs ASM

You only need ASM for a small part of your OS — but that part is impossible to do in C. Here's where ASM is required:

Bootloader
The very first code that runs. No C runtime exists yet. You must write this in ASM.
CPU Setup
Switching from 16-bit real mode → 64-bit long mode. Only ASM can do this.
Interrupts
When hardware (keyboard, timer) fires an interrupt, ASM saves/restores CPU state.
Context Switch
Switching between running programs. You must save/restore all registers manually.
Special Registers
Registers like CR0, CR3, GDT — C cannot touch these. ASM can.
The ratio: Your OS will be roughly 5% ASM and 95% C. Learn just enough ASM to do these jobs, then write everything else in C like you already know how to.
Chapter 02

Registers

Registers are tiny storage slots inside the CPU itself. Way faster than RAM. Think of them as the CPU's hands — it can only work with data that's in a register.

In x86_64 the main registers are 64-bit wide. Here are all the ones you need:

General Purpose Registers

RegisterNameWhat it's used for
raxAccumulatorMath results, return values from functions
rbxBaseGeneral purpose, often holds a base address
rcxCounterLoop counters, shift amounts
rdxDataI/O, extra data for multiply/divide
rsiSource IndexSource pointer in memory operations
rdiDest IndexDestination pointer, first function argument
rspStack PointerPoints to the top of the stack. Very important!
rbpBase PointerPoints to the base of the current stack frame
r8–r15ExtraExtra general purpose registers (x86_64 added these)

Register Sizes — Same Register, Different Sizes

This trips people up. One register has 4 names depending on how many bits you want to use:

register sizesconcept
rax   = 64-bit  (the full thing)
eax   = 32-bit  (lower half of rax)
ax    = 16-bit  (lower 16 bits)
al    = 8-bit   (lowest byte)
ah    = 8-bit   (second byte)

; For OS dev in 64-bit mode, you'll mostly use rax, rbx etc.
; But knowing eax/ax exists is important for legacy code

Special Registers

RegisterWhat it does
ripInstruction Pointer — points to the next instruction. You can't write to it directly
rflagsFlags register — stores results of comparisons (zero? negative? overflow?)
cr0Control Register — bit 0 enables protected mode. You'll use this in your bootloader
cr3Page table base address — used for virtual memory
Chapter 03

Memory & Addressing

Square brackets [ ] mean "go to this memory address and read/write there". Without brackets you're working with the value directly. With brackets you're working with what's at that address.

addressing modesx86_64 NASM
mov rax, 42          ; rax = 42  (immediate value)
mov rax, rbx         ; rax = rbx (register to register)
mov rax, [rbx]       ; rax = memory at address stored in rbx
mov rax, [0x1000]    ; rax = memory at address 0x1000
mov rax, [rbx+8]     ; rax = memory at address rbx + 8
mov rax, [rbx+rcx*4] ; rax = memory at rbx + rcx*4 (array access!)

Data Sizes in Memory

size specifiersNASM
mov byte  [rax], 5    ; write 1 byte  to memory
mov word  [rax], 5    ; write 2 bytes to memory
mov dword [rax], 5    ; write 4 bytes to memory
mov qword [rax], 5    ; write 8 bytes to memory
Chapter 04

MOV — Moving Data

mov is the most used instruction in ASM. It copies data from one place to another. It's the = of assembly.

Rule: You cannot MOV directly from memory to memory. You must go through a register. Load → Modify → Store.
mov examplesx86_64 NASM
; ✅ Valid MOVs
mov rax, 100          ; rax = 100
mov rbx, rax          ; rbx = rax
mov [rax], rbx        ; memory[rax] = rbx
mov rcx, [rbx]        ; rcx = memory[rbx]

; ❌ Invalid — can't go memory to memory directly
; mov [rax], [rbx]   ← this will NOT compile

; ✅ Do this instead:
mov rdx, [rbx]        ; load from memory into register
mov [rax], rdx        ; then store to other memory

LEA — Load Effective Address

Like MOV but gives you the address itself, not the value at that address. Super useful for pointer arithmetic.

lea examplex86_64 NASM
lea rax, [rbx+16]     ; rax = rbx + 16  (the ADDRESS, not the value there)
; In C this is:  rax = &array[4]  (if elements are 4 bytes)

; You can also use LEA for fast multiplication!
lea rax, [rbx*3]       ; rax = rbx * 3  (faster than using mul)
Chapter 05

Arithmetic

arithmetic instructionsx86_64 NASM
; ADD and SUB
add rax, 10        ; rax = rax + 10
add rax, rbx       ; rax = rax + rbx
sub rax, 5         ; rax = rax - 5
inc rax            ; rax++
dec rax            ; rax--

; MULTIPLY (unsigned)
mov rax, 6
mov rbx, 7
mul rbx            ; rdx:rax = rax * rbx  (result in rax, overflow in rdx)

; DIVIDE (unsigned)
mov rax, 100
mov rdx, 0         ; MUST zero rdx before div!
mov rbx, 7
div rbx            ; rax = quotient, rdx = remainder

; NEG — negate (make negative)
neg rax            ; rax = -rax
Watch out with DIV: Before dividing, always zero out rdx. If you forget, you'll get a divide error and your OS will crash. Classic beginner bug.
Chapter 06

Logic & Bit Operations

In OS dev you manipulate flags and bits constantly. Turning a feature on means setting bit 3 of some register. Checking if a device is ready means testing bit 7. This is where bitwise ops shine.

bitwise instructionsx86_64 NASM
and rax, rbx        ; rax = rax AND rbx  (mask bits)
or  rax, rbx        ; rax = rax OR  rbx  (set bits)
xor rax, rbx        ; rax = rax XOR rbx  (flip bits)
not rax             ; rax = NOT rax       (invert all bits)
shl rax, 1          ; rax = rax << 1      (shift left = multiply by 2)
shr rax, 1          ; rax = rax >> 1      (shift right = divide by 2)

; ---- Real OS Dev examples ----

; Set bit 0 of cr0 (enables protected mode)
or  eax, 1

; Clear bit 2 (mask it out)
and rax, 0xFFFFFFFB  ; 0xFFFFFFFB = all 1s except bit 2

; XOR a register with itself = zero it out (fastest way!)
xor rax, rax         ; rax = 0
Pro tip: xor rax, rax is the fastest way to zero a register. You'll see this everywhere in OS code.
Chapter 07

Jumps & Comparisons

This is how you do if statements and loops in ASM. First you compare two values, then you jump based on the result.

cmp and jumpsx86_64 NASM
; CMP sets flags but doesn't change registers
; It's basically SUB without saving the result

cmp rax, 10           ; compare rax with 10
je  .equal            ; jump if equal      (rax == 10)
jne .not_equal        ; jump if not equal  (rax != 10)
jl  .less             ; jump if less than  (rax < 10)
jg  .greater          ; jump if greater    (rax > 10)
jle .less_eq          ; jump if <=
jge .greater_eq       ; jump if >=
jmp .somewhere        ; unconditional jump (like goto)

; ---- This is an if/else in ASM ----
cmp rax, 0
je  .is_zero
; code for rax != 0 goes here
jmp .done
.is_zero:
; code for rax == 0 goes here
.done:

A Loop in ASM

loop examplex86_64 NASM
; for (rcx = 0; rcx < 10; rcx++)
xor rcx, rcx          ; rcx = 0
.loop_start:
cmp rcx, 10
jge .loop_end          ; exit if rcx >= 10

  ; your loop body goes here

  inc rcx               ; rcx++
  jmp .loop_start
.loop_end:
Chapter 08

The Stack

The stack is a region of memory that works like a stack of plates. You push things on top, you pop them off. The CPU manages it automatically with rsp (stack pointer). The stack grows downward in memory.

push & popx86_64 NASM
push rax    ; rsp -= 8, then memory[rsp] = rax
push rbx    ; rsp -= 8, then memory[rsp] = rbx
push rcx    ; rsp -= 8, then memory[rsp] = rcx

pop  rcx    ; rcx = memory[rsp], then rsp += 8
pop  rbx    ; gets rbx back
pop  rax    ; gets rax back  (LIFO order!)

; Push and pop must always be balanced!
; If you push 3 times you must pop 3 times
Stack alignment: In x86_64, before calling any function, rsp must be 16-byte aligned. If you call C code from ASM and it crashes mysteriously, check your alignment first.
Chapter 09

Functions & The Calling Convention

When your ASM calls C code (or C calls ASM), both sides must agree on which registers hold the arguments and where the return value goes. This agreement is called the System V AMD64 calling convention — the standard on Linux/most OS dev.

The Rules

RegisterRole
rdi1st argument
rsi2nd argument
rdx3rd argument
rcx4th argument
r85th argument
r96th argument
raxReturn value
rbx, rbp, r12–r15Callee-saved (you must restore these if you use them)
rax, rcx, rdx, rsi, rdi, r8–r11Caller-saved (can be destroyed by called function)
calling C from ASMx86_64 NASM
; Calling this C function:  int add(int a, int b)
mov  rdi, 10        ; first argument  = 10
mov  rsi, 20        ; second argument = 20
call add            ; call the function
; result is now in rax  (rax = 30)

; Writing a simple function in ASM
my_function:
    push rbp            ; save old base pointer
    mov  rbp, rsp       ; set new base pointer

    ; your code here
    ; first arg is in rdi
    ; put return value in rax

    pop  rbp            ; restore base pointer
    ret                  ; return (pops rip from stack)
Chapter 10

Interrupts

An interrupt is when hardware (or software) says "stop what you're doing, handle this now!". Your keyboard press? Interrupt. Timer tick? Interrupt. Page fault? Interrupt. Your OS must handle all of these.

interrupt handler stubx86_64 NASM
; When an interrupt fires, the CPU jumps here
; You MUST save ALL registers before doing anything
isr_handler:
    push rax          ; save all registers
    push rbx
    push rcx
    push rdx
    push rsi
    push rdi
    push rbp
    push r8
    push r9
    push r10
    push r11

    call handle_interrupt  ; call your C function

    pop  r11          ; restore all registers (reverse order!)
    pop  r10
    pop  r9
    pop  r8
    pop  rbp
    pop  rdi
    pop  rsi
    pop  rdx
    pop  rcx
    pop  rbx
    pop  rax
    iretq             ; return from interrupt (special return!)
Chapter 11

Your Bootloader in ASM

This is the first code your OS ever runs. The BIOS loads the first 512 bytes of your disk into memory at address 0x7C00 and jumps to it. That's your moment.

minimal 16-bit bootloaderx86_64 NASM
[bits 16]               ; BIOS starts us in 16-bit mode
[org 0x7C00]            ; we're loaded at this address

start:
    cli                 ; disable interrupts while setting up
    xor ax, ax
    mov ds, ax          ; zero out segment registers
    mov es, ax
    mov ss, ax
    mov sp, 0x7C00      ; set up stack below us
    sti                 ; re-enable interrupts

    ; Print a character using BIOS interrupt
    mov  ah, 0x0E       ; BIOS teletype function
    mov  al, '!'        ; character to print
    int  0x10           ; call BIOS video interrupt

    jmp  $              ; loop forever (for now)

; Pad to 512 bytes and add boot signature
times 510-($-$$) db 0
dw 0xAA55               ; magic number — BIOS checks for this
Those last 2 bytes 0xAA55 are the boot signature. If they're not there, the BIOS refuses to boot your code. This is the most important 2 bytes you'll ever write.
Chapter 12

GDT & Switching to 64-bit

The GDT (Global Descriptor Table) is a data structure you put in memory that tells the CPU about memory segments. To get to 64-bit long mode, you must: 1) set up a GDT, 2) load it with the lgdt instruction, 3) set bit 0 of CR0, 4) far jump.

loading the GDTx86_64 NASM
; Load the GDT descriptor
lgdt [gdt_descriptor]

; Set bit 0 of CR0 — this enables protected mode!
mov eax, cr0
or  eax, 1
mov cr0, eax

; Far jump to flush the pipeline and enter 32-bit mode
jmp 0x08:protected_mode  ; 0x08 = code segment selector

[bits 32]
protected_mode:
    ; Now in 32-bit protected mode
    ; Next step: enable long mode (64-bit)
    ; That involves CR4, EFER MSR, and page tables
    ; — the OSDev wiki covers this step by step
Chapter 13

Handing Off to C

Once your ASM has set up the CPU and memory, you call your C kernel. This is the moment your tiny 5% ASM hands the world to your 95% C code.

jumping to your C kernelx86_64 NASM
[bits 64]
long_mode_start:
    ; Zero out all segment registers
    mov ax, 0
    mov ss, ax
    mov ds, ax
    mov es, ax
    mov fs, ax
    mov gs, ax

    ; Call your C kernel entry point!
    call kernel_main

    ; If kernel_main ever returns, halt the CPU
    hlt
your C kernel starts hereC
// kernel.c
void kernel_main() {
    // You're in 64-bit mode. You own the machine.
    // Write to VGA text buffer to print to screen
    char* vga = (char*)0xB8000;
    vga[0] = 'H';
    vga[1] = 0x0F;  // white on black
    while(1);       // halt
}
That VGA buffer at 0xB8000 — write a character + a color byte there and it appears on screen. No drivers needed. That's your first "Hello World" from your OS.
Chapter 14

NASM Setup

NASM (Netwide Assembler) is the assembler you'll use. It's free, it's everywhere, and it's what most OS dev tutorials use.

install & compilebash
# Install NASM
sudo apt install nasm          # Ubuntu/Debian
brew install nasm              # macOS

# Assemble your bootloader
nasm -f bin bootloader.asm -o bootloader.bin

# Test it in QEMU (a free CPU emulator — no real hardware needed!)
sudo apt install qemu-system-x86
qemu-system-x86_64 -drive format=raw,file=bootloader.bin

# Assemble ASM that will link with C
nasm -f elf64 boot.asm -o boot.o
gcc -c kernel.c -o kernel.o -ffreestanding -nostdlib
ld -T linker.ld boot.o kernel.o -o kernel.elf
QEMU is your best friend. You can boot and test your OS in a virtual machine without touching real hardware. Mess up? Just restart QEMU. No risk, infinite experiments.

Your Next Steps

You now know enough ASM to start. Here's what to do next, in order:

Step 1
Install NASM and QEMU. Write the bootloader above, boot it, see the '!' appear on screen.
Step 2
Read the OSDev Wiki — osdev.org. Start with "Bare Bones" tutorial. It's the bible.
Step 3
Get into 32-bit protected mode. Follow the GDT and A20 line guides on OSDev.
Step 4
Get into 64-bit long mode. Set up basic paging.
Step 5
Call kernel_main() in C. From here, write everything in C — memory allocator, drivers, scheduler.
Step 6
Read "Writing a Simple OS from Scratch" by Nick Blundell — free PDF. Best beginner OS book.
