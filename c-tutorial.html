C Programming

The language your OS will be written in. Learn it right, learn it deep.

Step 1 of 2 OS Dev path Free forever

What is C?

C is the closest you can get to the hardware without writing Assembly. There's no garbage collector, no runtime managing memory for you. You control everything. That's exactly what OS development needs.

Linux, Windows, macOS — all written mostly in C. Your OS will be too.

Fast
C compiles directly to machine code. No virtual machine, no interpreter. Raw speed.
Low level
You can read and write any memory address. Talk directly to hardware. Control every byte.
Portable
The same C code runs on x86, ARM, RISC-V. Your OS skills transfer everywhere.
Minimal
The language is tiny. No built-in classes, no bloat. You build everything from scratch.

Chapter 02

Variables & Types

C is statically typed — every variable has a fixed type and you must declare it. The types map directly to sizes in memory.

TypeSizeWhat it holds
char1 byteA single character or small integer (-128 to 127)
int4 bytesA whole number (-2 billion to 2 billion)
long8 bytesA bigger whole number
float4 bytesDecimal number (less precise)
double8 bytesDecimal number (more precise)
void—Nothing / no type. Used for functions that return nothing

Fixed-width types — what OS dev actually uses

In OS dev you need to know exactly how many bytes a variable takes. Use these instead of plain int:

os dev typesC
#include <stdint.h>   // gives us fixed-width types

uint8_t   a = 255;      // 1 byte,  0 to 255
uint16_t  b = 65535;    // 2 bytes, 0 to 65535
uint32_t  c = 100000;   // 4 bytes
uint64_t  d = 999999;   // 8 bytes

int8_t    e = -10;      // signed 1 byte, -128 to 127
int32_t   f = -50000;   // signed 4 bytes

// In OS dev you'll use uint8_t, uint16_t, uint32_t, uint64_t constantly

Declaring variables

variablesC
int x = 10;           // declare and initialize
int y;                // declare only (value is undefined!)
y = 20;               // assign later

int a, b, c;          // multiple variables same type
a = b = c = 0;        // set all to 0

const int MAX = 100;  // constant — cannot change
Watch out: Uninitialized variables in C contain random garbage values — whatever bytes were in that memory before. Always initialize your variables, especially in OS code where bugs can crash the whole machine.
Chapter 03

Operators

all operatorsC
// Arithmetic
int a = 10 + 5;    // 15  — add
int b = 10 - 5;    // 5   — subtract
int c = 10 * 5;    // 50  — multiply
int d = 10 / 3;    // 3   — divide (integer, drops remainder)
int e = 10 % 3;    // 1   — modulo (remainder)

// Shorthand
a += 5;   // a = a + 5
a -= 2;   // a = a - 2
a++;      // a = a + 1
a--;      // a = a - 1

// Comparison (returns 1 for true, 0 for false)
a == b    // equal
a != b    // not equal
a <  b    // less than
a >  b    // greater than
a <= b    // less or equal
a >= b    // greater or equal

// Logical
a && b    // AND — true if both are true
a || b    // OR  — true if at least one is true
!a        // NOT — flips true/false
Chapter 04

If / Else

if / else if / elseC
int score = 85;

if (score >= 90) {
    printf("A grade\n");
} else if (score >= 70) {
    printf("B grade\n");   // this runs
} else {
    printf("Try again\n");
}

// Switch statement — cleaner for many fixed cases
int day = 3;
switch (day) {
    case 1: printf("Monday\n");    break;
    case 2: printf("Tuesday\n");   break;
    case 3: printf("Wednesday\n"); break; // this runs
    default: printf("Other\n");    break;
}
Don't forget break! In a switch statement, if you leave out break, execution falls through to the next case. This causes hard-to-find bugs.
Chapter 05

Loops

for / while / do-whileC
// FOR loop — when you know how many times
for (int i = 0; i < 10; i++) {
    printf("%d\n", i);   // prints 0 through 9
}

// WHILE loop — when you don't know how many times
int x = 0;
while (x < 5) {
    printf("%d\n", x);
    x++;
}

// DO-WHILE — runs at least once
do {
    printf("runs at least once\n");
    x++;
} while (x < 3);

// Infinite loop — used everywhere in OS dev
while (1) {
    // kernel idle loop — runs forever
}

// break and continue
for (int i = 0; i < 10; i++) {
    if (i == 5) break;     // exit loop entirely
    if (i == 3) continue;  // skip this iteration
    printf("%d\n", i);
}
Chapter 06

Functions

Functions in C must be declared before they're used. The format is: return_type name(parameters).

functionsC
// A function that adds two numbers and returns the result
int add(int a, int b) {
    return a + b;
}

// A function that returns nothing (void)
void print_hello() {
    printf("Hello!\n");
}

// Calling functions
int main() {
    int result = add(3, 7);   // result = 10
    print_hello();             // prints Hello!
    return 0;
}

// Forward declaration — declare before defining
int multiply(int a, int b);   // just the signature

int main() {
    multiply(4, 5);             // works even though defined below
    return 0;
}

int multiply(int a, int b) {   // actual definition
    return a * b;
}
Chapter 07

Arrays

An array is a block of memory holding multiple values of the same type, side by side. Arrays are zero-indexed — the first element is at index 0.

arraysC
// Declare an array of 5 ints
int numbers[5] = {10, 20, 30, 40, 50};

numbers[0]   // = 10  (first element)
numbers[4]   // = 50  (last element)
numbers[5]   // ❌ DANGER — out of bounds! Undefined behavior

// Loop through an array
for (int i = 0; i < 5; i++) {
    printf("%d\n", numbers[i]);
}

// Array of chars = a string
char name[6] = {'H', 'e', 'l', 'l', 'o', '\0'};
//                                              ↑ null terminator — marks the end

// 2D array (like a grid)
int grid[3][3] = {
    {1, 2, 3},
    {4, 5, 6},
    {7, 8, 9}
};
grid[1][2]  // = 6  (row 1, column 2)
Chapter 08

Pointers

Pointers are the most important concept in C — and the one most people struggle with. A pointer is simply a variable that holds a memory address instead of a value.

In OS dev you use pointers constantly — to talk to hardware registers, to manage memory, to build data structures.

pointers basicsC
int x = 42;
int* p = &x;     // p holds the ADDRESS of x
                  // & means "address of"

printf("%d\n", x);    // prints 42   (the value)
printf("%p\n", p);    // prints address like 0x7fff5abc
printf("%d\n", *p);   // prints 42   (* means "value at this address")

*p = 100;             // change x through the pointer
printf("%d\n", x);    // now prints 100!

Pointers and OS Dev Hardware

hardware pointersC
// In OS dev, hardware is at fixed memory addresses.
// You just point at the address and write to it.

// VGA text buffer — writing here puts text on screen
uint16_t* vga = (uint16_t*)0xB8000;
vga[0] = 0x0F00 | 'H';   // write 'H' in white on black
vga[1] = 0x0F00 | 'i';   // write 'i'

// That's it. No library. No function call. Just a pointer write.

Pointer Arithmetic

pointer arithmeticC
int arr[5] = {10, 20, 30, 40, 50};
int* p = arr;    // p points to arr[0]

*p       // = 10  (arr[0])
*(p+1)   // = 20  (arr[1]) — pointer moves by sizeof(int) = 4 bytes
*(p+2)   // = 30  (arr[2])

p++;     // move pointer to next element
*p       // = 20 now
Key insight: An array name in C is a pointer to the first element. arr[2] and *(arr+2) are identical. This is why pointers and arrays feel the same in C.
Chapter 09

Heap Memory

Variables declared normally live on the stack — they disappear when the function returns. If you need memory that persists longer, you allocate it on the heap using malloc.

malloc & freeC
#include <stdlib.h>   // for malloc and free

// Allocate space for 10 ints on the heap
int* arr = (int*) malloc(10 * sizeof(int));

if (arr == NULL) {
    // malloc failed — always check!
    return -1;
}

arr[0] = 100;   // use it just like a normal array
arr[1] = 200;

free(arr);      // ALWAYS free when done — or you leak memory
arr = NULL;     // good habit — null the pointer after freeing
In OS dev: There's no malloc in your kernel until YOU write one. That's actually one of the first things you'll build — a memory allocator. This is where it gets fun.
Chapter 10

Strings

In C there's no built-in string type. A string is just an array of char ending with a null terminator '\0'. That zero byte tells functions where the string ends.

strings in CC
// String literal — stored in read-only memory
char* msg = "Hello";   // = {'H','e','l','l','o','\0'}

// Char array — stored on the stack, can be modified
char buf[6] = "Hello";
buf[0] = 'h';   // ok — modifiable

// String length — count until '\0'
#include <string.h>
int len = strlen("Hello");   // = 5 (doesn't count the \0)

// Writing your own strlen — this is what OS dev feels like
int my_strlen(char* s) {
    int i = 0;
    while (s[i] != '\0') i++;
    return i;
}
Chapter 11

Structs

A struct groups related variables together under one name. In OS dev you'll use structs for everything — process control blocks, memory descriptors, driver state, interrupt frames.

structsC
// Define a struct
struct Process {
    uint32_t  pid;        // process ID
    uint64_t  stack_ptr;  // stack pointer
    char      name[32];   // process name
    uint8_t   state;      // running, sleeping, etc
};

// Create and use a struct
struct Process p;
p.pid       = 1;
p.stack_ptr = 0x7FFF0000;
p.state     = 1;

// typedef — cleaner syntax (no need to write 'struct' each time)
typedef struct {
    uint32_t pid;
    uint8_t  state;
} Process;

Process p2;   // much cleaner!
p2.pid = 2;

// Pointer to struct — use arrow (->) instead of dot (.)
Process* ptr = &p2;
ptr->pid     = 3;    // same as (*ptr).pid = 3
ptr->state   = 0;
Chapter 12

Enums

Enums give names to numbers. Instead of remembering that state 0 means "running" and state 1 means "sleeping", you write the words directly.

enumsC
typedef enum {
    PROCESS_RUNNING  = 0,
    PROCESS_SLEEPING = 1,
    PROCESS_DEAD     = 2
} ProcessState;

ProcessState state = PROCESS_RUNNING;

if (state == PROCESS_RUNNING) {
    // much clearer than:  if (state == 0)
}
Chapter 13

Headers & Includes

Header files (.h) contain declarations — function signatures, struct definitions, constants. Source files (.c) contain the actual code. This is how you split a big project across multiple files.

kernel.hC
// kernel.h — declarations only
#ifndef KERNEL_H    // include guard — prevents double-including
#define KERNEL_H

#include <stdint.h>

void     kernel_main();
void     print_char(char c, uint8_t color);
uint32_t get_tick();

#define VGA_ADDRESS 0xB8000
#define VGA_WHITE   0x0F

#endif
kernel.cC
// kernel.c — actual implementations
#include "kernel.h"   // your own header — use quotes

void print_char(char c, uint8_t color) {
    uint16_t* vga = (uint16_t*) VGA_ADDRESS;
    vga[0] = (color << 8) | c;
}

void kernel_main() {
    print_char('!', VGA_WHITE);
    while(1);
}
Chapter 14

Freestanding C

Normal C programs run on top of an OS which provides printf, malloc, and all the standard library. In OS dev there is no OS underneath you. You compile in "freestanding" mode.

freestanding compile flagsbash
gcc -ffreestanding   # no standard library
    -nostdlib        # no startup files
    -nostdinc        # no standard headers
    -fno-builtin     # no built-in functions
    -m64             # 64-bit output
    kernel.c -o kernel.o
what you CAN still useC
// These headers work in freestanding mode:
#include <stdint.h>    // uint8_t, uint32_t etc
#include <stddef.h>    // NULL, size_t
#include <stdbool.h>   // bool, true, false

// Everything else — you write yourself.
// No printf. No malloc. No memcpy.
// That's the point. You own everything.
Chapter 15

Bit Operations

In OS dev you manipulate hardware registers one bit at a time. A single byte controls 8 on/off features. You need to set, clear, and test individual bits without touching the others.

bit manipulationC
uint8_t flags = 0b00000000;   // all bits off

// SET bit 3 (turn it ON)
flags |= (1 << 3);    // flags = 0b00001000

// CLEAR bit 3 (turn it OFF)
flags &= ~(1 << 3);   // flags = 0b00000000

// TOGGLE bit 3 (flip it)
flags ^= (1 << 3);    // flips bit 3

// TEST bit 3 (is it on?)
if (flags & (1 << 3)) {
    // bit 3 is set
}

// Real example — VGA color byte
// bits [7:4] = background color, bits [3:0] = foreground color
uint8_t color = (0 << 4) | 15;   // black bg, white fg = 0x0F
Chapter 16

Volatile & Memory-Mapped I/O

The compiler is smart — it caches values in registers to avoid reading memory repeatedly. But hardware registers change on their own. You must use volatile to tell the compiler "always read this from memory, never cache it."

volatileC
// WITHOUT volatile — compiler might optimize this to only read once
uint32_t* status_reg = (uint32_t*)0xFEE00000;
while (*status_reg == 0);   // might loop forever — compiler cached 0!

// WITH volatile — compiler reads from address every single time
volatile uint32_t* status_reg = (volatile uint32_t*)0xFEE00000;
while (*status_reg == 0);   // correctly waits for hardware to change it

// Rule: any pointer to a hardware register should be volatile
This is one of the most common OS dev bugs. Code looks correct, but the compiler "optimizes" away your hardware reads. When in doubt with hardware — use volatile.
